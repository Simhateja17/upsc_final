Auth flow:
1. App calls Supabase Auth REST API to login/signup
2. Supabase returns access_token + refresh_token
3. App stores tokens securely
4. App calls backend `/auth/callback` with tokens to sync/create user
5. All subsequent API calls include `Authorization: Bearer {access_token}`
6. Backend verifies JWT via Supabase JWKS

    Create the authentication layer and wire it to LoginScreen.

    **SupabaseAuthService:**
    - Uses raw Ktor client (not ApiClient) to call Supabase Auth REST API directly
    - Base URL: `https://pjbijjhrxlkotldenydv.supabase.co/auth/v1`
    - Headers: `apikey: {anon_key}`, `Content-Type: application/json`
    - Methods:
      - `login(email: String, password: String): AuthTokens` → POST `/token?grant_type=password` with body `{ email, password }`, parse response to extract `access_token`, `refresh_token`, `expires_in`
      - `signup(email: String, password: String, firstName: String?, lastName: String?): AuthTokens` → POST `/signup` with body `{ email, password, data: { first_name, last_name } }`
      - `refreshToken(refreshToken: String): AuthTokens` → POST `/token?grant_type=refresh_token` with body `{ refresh_token }`
      - `logout()` → POST `/logout` with Bearer token
    - Parse Supabase Auth responses using kotlinx.serialization

    **AuthRepository:**
    - Depends on: SupabaseAuthService, ApiClient, TokenStorage
    - Methods:
      - `suspend fun login(email: String, password: String): Result<User>`
        - Calls SupabaseAuthService.login
        - Stores tokens in TokenStorage
        - Calls backend `/auth/callback` via ApiClient.post with `{ accessToken, refreshToken }`
        - Calls backend `/auth/me` via ApiClient.get to get user profile
        - Returns User on success
      - `suspend fun signup(email: String, password: String, firstName: String?, lastName: String?): Result<User>` (similar flow)
      - `suspend fun logout(): Result<Unit>`
        - Calls backend `/auth/logout` then Supabase logout
        - Clears TokenStorage
      - `suspend fun getCurrentUser(): User?`
        - Returns cached user or fetches from `/auth/me` if token exists
      - `fun isLoggedIn(): Boolean` (checks token existence)

    **User model:**
    ```kotlin
    @Serializable
    data class User(
        val id: String,
        val email: String,
        val firstName: String? = null,
        val lastName: String? = null,
        val avatarUrl: String? = null,
        val role: String = "user",
        val isActive: Boolean = true
    )
    ```

    **AuthViewModel:**
    - State class: `AuthUiState` with `isLoading: Boolean`, `user: User?`, `error: String?`
    - Methods: `login(email, password)`, `signup(email, password, firstName, lastName)`, `logout()`
    - Uses `viewModelScope` with ` kotlinx.coroutines ` 
    - Exposes `uiState: StateFlow<AuthUiState>`

    **ServiceLocator:**
    - Manual DI container (object)
    - Lazily initializes: TokenStorage, ApiClient, SupabaseAuthService, AuthRepository
    - Provides factory/getter for AuthViewModel
    - Document clearly that this is a temporary solution; Koin may be introduced later

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Spoofing | LoginScreen | mitigate | Use HTTPS for Supabase and backend; validate SSL certificates |
| T-01-02 | Tampering | TokenStorage.android | mitigate | Use EncryptedSharedPreferences with AES256_SIV + AES256_GCM |
| T-01-03 | Information Disclosure | TokenStorage | mitigate | Tokens never logged; clear on logout; iOS uses keychain |
| T-01-04 | Denial of Service | ApiClient | mitigate | Request timeout (15s); circuit breaker pattern for retries |
| T-01-05 | Elevation of Privilege | AuthInterceptor | mitigate | Only reads token from secure storage; never accepts token from UI |
</threat_model>
